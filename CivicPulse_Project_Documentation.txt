1. PROJECT OVERVIEW
================================================================================

CivicPulse is a full-stack AI-powered civic complaint management platform that
enables citizens to report municipal issues (potholes, water leaks, garbage, etc.)
and allows government authorities to manage, track, and resolve these complaints
efficiently. The platform uses multiple AI systems (Google Gemini NLP + YOLOv8
Computer Vision) to automatically categorize, prioritize, and analyze reported
issues.

Project Name     : CivicPulse – Smart Civic Intelligence
Type             : Full-Stack Web Application + AI Microservice
Target Users     : Citizens and Municipal/Government Authorities
Domain           : Smart City / E-Governance / Civic Tech

4. SYSTEM WORKFLOW
================================================================================

----- 4.1 CITIZEN COMPLAINT SUBMISSION WORKFLOW -----

Step 1: Citizen opens the "Report Issue" page
Step 2: GPS auto-fetches current location using:
        → Browser Geolocation API (primary)
        → ipapi.co IP-based fallback (if GPS denied)
Step 3: Reverse geocoding converts coordinates to human-readable address:
        → ArcGIS Geocoding API (primary)
        → Nominatim/OpenStreetMap (fallback)
Step 4: Citizen fills in title and description
        → Can use voice input via Web Speech API (speech-to-text)
        → Real-time AI category suggestion as they type (keyword matching)
Step 5: Citizen optionally selects a category from 10 categories
Step 6: Citizen optionally uploads images (photos of the issue)
        → Images uploaded to Cloudinary CDN
Step 7: Citizen clicks "Submit"
Step 8: Gemini AI analyzes the complaint text:
        → Determines category, priority, severity score (0.0-1.0)
        → Generates an executive summary for authorities
Step 9: If images were uploaded, the YOLO AI microservice is called:
        → Downloads the image
        → Runs YOLOv8s-World zero-shot detection
        → Detects civic issues (pothole, garbage, broken traffic light, etc.)
        → Maps detected class to priority level
        → Updates Firestore with AI vision results
Step 10: Complaint is saved to Firestore with all AI-enhanced data
Step 11: A notification is created for the authority
Step 12: Duplicate detection runs (location + category proximity check)
Step 13: Success toast shown; citizen redirected to "My Complaints"


----- 4.2 AUTHORITY COMPLAINT MANAGEMENT WORKFLOW -----

Step 1: Authority logs into the platform
Step 2: Dashboard shows real-time stats:
        → Total, pending, in-progress, resolved, high-priority counts
        → Monthly complaint trend charts
        → Priority distribution pie chart
        → Category breakdown bar chart
        → Recent complaints table
Step 3: Authority navigates to "Manage Complaints"
Step 4: Authority can filter by: search text, status, priority, category, sort
Step 5: Authority clicks on a complaint to view the full detail modal:
        → Citizen info, description, location (with Google Maps link)
        → Attached media/photos
        → AI Analysis panel:
           - Severity prediction bar (0-100%)
           - AI-predicted category
           - AI Executive Summary (Gemini-generated)
           - YOLO Vision detection results (if image was uploaded)
           - Duplicate flag warning
Step 6: Authority takes action:
        → Mark as "In Progress" (submitted → in_progress)
        → Mark as "Resolved" (in_progress → resolved)
        → "Reject" (if invalid)
        → "Reopen" (if resolved/rejected needs re-examination)
        → Change priority (dropdown: high/medium/low)
        → Delete complaint
Step 7: Status change triggers:
        → Firestore update (status, timeline entry)
        → Notification created for the citizen
        → Toast confirmation


----- 4.3 NOTIFICATION WORKFLOW -----

Trigger 1: Citizen submits complaint → Authority gets "New Complaint" notification
Trigger 2: Authority changes status → Citizen gets "Complaint Updated" notification

Notifications are:
  → Stored in Firestore "notifications" collection
  → Fetched in real-time via onSnapshot listener
  → Displayed in bell icon dropdown in navbar
  → Show unread indicator (red dot)
  → Support mark-as-read and delete actions
  → Display relative timestamps (2m, 5h, 3d, 1w)

7. AI MICROSERVICE API ENDPOINTS
================================================================================

Base URL: http://localhost:8000

----- POST /api/predict_priority -----
Purpose  : Analyze a complaint image using YOLOv8s-World for issue detection
Request  : { "image_url": "https://...", "complaint_id": "CIV-001" }
Response : {
    "status": "success",
    "prediction": {
        "ai_detected_issue": "Pothole",
        "ai_priority": "High",
        "ai_confidence": 0.87,
        "ai_details": "pothole (0.87)"
    }
}
Side Effect: Updates the complaint document in Firestore with AI results

----- GET /api/export/complaints -----
Purpose  : Export cleaned complaint data as JSON (for Power BI integration)
Auth     : Requires X-API-KEY header
Params   : fromDate, toDate, category, status (all optional filters)
Features : Duplicate removal, null coordinate filtering, geospatial extraction

----- GET /api/export/complaints/csv -----
Purpose  : Export cleaned complaint data as CSV file download
Auth     : Requires X-API-KEY header
Params   : Same as JSON export
Response : Streaming CSV file with proper headers

9. KEY DESIGN PATTERNS & IMPLEMENTATION DETAILS
================================================================================

----- 9.1 STATE MANAGEMENT -----
  • React Context API for global state (no Redux/Zustand)
  • Three context providers wrapping the app:
    → AuthProvider: user session, login, register, logout, profile update
    → ComplaintProvider: complaint CRUD, real-time sync, stats computation
    → NotificationProvider: real-time notification listener, read/delete

----- 9.2 REAL-TIME DATA -----
  • Firestore onSnapshot listeners for complaints and notifications
  • Any change in Firestore (add/update/delete) instantly reflects in the UI
  • No need to refresh the page or re-fetch data manually

----- 9.3 GEOLOCATION STRATEGY -----
  Three-tier location detection with robust fallbacks:
  1. Browser Geolocation API (GPS/WiFi) → most accurate
  2. IP-based geolocation via ipapi.co → moderate accuracy
  3. Default coordinates (New Delhi) → last resort

  Two-tier reverse geocoding:
  1. ArcGIS Geocoding API → primary (detailed addresses)
  2. Nominatim/OpenStreetMap → fallback (open-source)

----- 9.4 AI FALLBACK STRATEGY -----
  If Gemini API fails or key is missing:
  → Local keyword-based categorization runs (text matching)
  → Default priority assigned based on keyword urgency
  → Fallback overview message generated

  If YOLO microservice is unavailable:
  → Complaint saved without vision analysis
  → Only Gemini NLP results are used

----- 9.5 DUPLICATE DETECTION -----
  Proximity-based duplicate check:
  → Compares new complaint's description keywords against existing complaints
  → Checks if location is within ~0.005 degrees (~500m) of existing complaint
  → If both category keyword and location match → flagged as potential duplicate
  → Visual warning shown to citizen and flagged for authority review

----- 9.6 UI/UX DESIGN -----
  • Dark theme with glassmorphism (backdrop-blur, semi-transparent cards)
  • Gradient accents (primary blue → secondary purple → accent green)
  • Smooth animations: fade-in, scale, transitions
  • Responsive grid layouts adapting to screen sizes
  • Status badges: color-coded (submitted=blue, in_progress=yellow,
    resolved=green, rejected=red)
  • Priority badges: color-coded (high=red, medium=yellow, low=green)

13. POWER BI INTEGRATION
================================================================================

The AI microservice includes Power BI export endpoints for advanced analytics:

  • JSON Export: GET /api/export/complaints
    → Returns cleaned, deduplicated complaint data
    → Supports date range, category, and status filters
    → Geospatial fields extracted for map visualizations
    → Protected by API key authentication

  • CSV Export: GET /api/export/complaints/csv
    → Streaming CSV download for direct import into Power BI
    → Same filtering and cleaning as JSON endpoint

Data Cleaning Pipeline:
  1. Remove duplicate-flagged complaints
  2. Remove document ID duplicates
  3. Apply date range filters
  4. Extract and validate geospatial coordinates
  5. Normalize area names and severity scores
  6. Map to clean, consistent schema
